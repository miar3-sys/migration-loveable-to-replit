Migration Assessment

Moving from Lovable + Supabase to Replit involves a few manual steps since the app is still an early prototype.

  • Export Code: Download the React code generated by Lovable.
  • Export Data: Run a SQL export (supabase-schema.sql) from Supabase to get the database structure and any superficial data.
  • Setup Replit: Create a new Replit project, upload the React code, and set up a Replit Postgres database.
  • Rewrite Auth: Supabase handles the @illinois.edu Google login automatically. On Replit, we will need to write custom backend code (like Auth.js or Passport.js) to handle Google logins.
  • Update Connections: Change the database connection strings in the code to point to Replit DB instead of Supabase.

Risk Summary

Before migrating IlliniXchange to Replit, consider these potential risks:

  • Loss of Visual Builder

    Lovable allows for rapid UI building without writing much code. Moving to Replit means all future UI updates (like adding new tabs or changing the profile page) must be coded manually or generated via text prompts, which might slow down design tweaks.

  • Authentication Complexity

    Supabase makes restricting logins to "@illinois.edu" very easy. Rebuilding this securely from scratch on Replit requires careful backend coding to ensure non-students can't access the trading platform.

  • All-in-One Dependency

    Currently, if Lovable goes down, the database (Supabase) is still safe. If Replit goes down, the code editor, the hosting, and the database all go down at the same time.

  • Scaling Costs

    While Replit is cheap to start, scaling a unified server for thousands of UIUC students can become more expensive than Supabase's generous free/low-cost tiers for database-heavy apps.
